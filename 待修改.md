# Hot-Monitor 系统交互与排产调试疑难解答

针对您记录在册的三个核心疑问以及近期终端出现的 SMTP 邮件发送报错（`Unexpected socket close` 与 `connect ETIMEDOUT 198.18.0.184:587`），以下是系统底层的详细运行机理解答。

---

## 🚨 特别追加：为什么邮件投递全盘失败？（代理劫持原理解析）

在终端中连续观察到以下两种错误：
1. `Error: Unexpected socket close`
2. `Error: connect ETIMEDOUT 198.18.0.184:587`

### 💡 核心病因：Clash / 代理软件的 TUN / Fake-IP 模式强行拦截
在连接超时报错中出现的 IP 地址 `198.18.0.184` 是非常有代表性的 **Clash TUN / Fake-IP 模式所分配的虚拟网络 IP 段**。
- **劫持发生**：后端服务发往发信服务器（如 `smtp.qq.com`）的 TCP 587/465 协议请求，直接被本地代理软件（如 Clash）的网卡驱动拦截，并强行接管到其 Fake-IP 代理池中。
- **协议冲突**：代理软件擅长处理 HTTP/HTTPS 协议流量。但 SMTP 协议属于发信专用的纯 TCP 握手。代理软件对其分流或转发不当，会导致握手包被强制中断（`Unexpected socket close`），或者直接卡死在虚拟网关层导致超时（`connect ETIMEDOUT`）。
- **一键解法**：
  - **方法一**：关闭本地代理软件（如 Clash / v2ray）的 **TUN 模式** 或 **系统代理**。
  - **方法二**：在代理软件的配置文件中，将 SMTP 相关的域名（如 `smtp.qq.com`, `smtp.gmail.com`）加入到 `DIRECT`（直连）规则或 Bypass（绕过代理）黑名单中。

## 📝 待修改及设计疑难官方解答

### 1. 目标邮箱无法输入多个的问题

* **当前现状**：这是因为前端创建/编辑智能订阅规则时，输入框对字符分割处理不够友好，导致难以通过简单的界限符添加多个接收者。
* **改进方案**：
  * 后端接口（`repositories.ts`）在 `SubscriptionRuleRecord` 的字段中已经将 `recipients`（接收邮箱）设计为 `text[]` (字符串数组) 格式。
  * 我们后续可以通过优化前端，将目标邮箱输入框升级为支持使用**英文逗号 `,`** 或**空格、换行**来自由切分出多个邮箱地址，在提交表单时以数组形式传给后端，实现多目标邮箱投递。

---

### 2. 勾选了关键词，但未打开任务开关/手动扫描，能否成功触发？

* **官方解答**：**绝对不能。**
* **底层数据链传递原理解析**：
  ```
  [ 监控任务 (Monitors) ] (需为开启状态)
         │
         ▼ (通过轮询或手动扫描触发抓取)
  [ 多源原始事件 (Events) ]
         │
         ▼ (经 AI 真实度过滤与聚合分类)
  [ 热点聚合簇 (Hotspots) ] (包含 accepted 状态的高价值情报)
         │
         ▼ (路由过滤分流器依据关键词检索)
  [ 订阅路由规则 (Subscription Rules) ] 
         │
         ▼ (通过 SMTP 通道)
  [ 目标投递邮箱 (Recipients) ]
  ```
* **核心结论**：订阅分流是一套“下游”过滤器，它必须依赖“上游”的采集流程产生原材料。如果任务管理的开关为 `off`，且不触发手动扫描，上游就**不会产生任何新数据**流入 `events` 和 `hotspots` 表，下游的订阅规则也就无从匹配，因此不会产生任何投递。要成功运行，上游任务开关必须为 `on`。

---

### 3. 发送信息时的邮件信息是如何整理的？

当上游成功捕获并由 AI 判定生成高价值热点簇（`hotspots`）后，`NotificationService`（通知分发模块）会按照以下流程对邮件信息进行整理与打包：

1. **多源数据整合**：从数据库中捞取该热点关联的所有真实原始事件（如 Twitter/X 发言、官方 RSS 博客内容、知乎热帖），把互动的点赞量、转发数等汇总。
2. **AI 情报总结**：提取 AI 服务分析整理出的**核心要点摘要**（`summary`）和推荐置信度评分（`score`）。
3. **HTML 模板渲染**：将上述数据注入预制的 HTML 模板中，渲染出排版美观且具有良好视觉阶梯的邮件网页，其中包含：
   * 📌 热点核心简报摘要。
   * 📊 推荐度、多样性及置信度评分。
   * 🔗 关联的原始佐证链接列表（供点击追踪溯源）。
   * 👍/👎 邮件底部的用户反馈赞踩按钮（以便回传反馈调优 AI 模型）。
4. **底层投递**：最终，组装完成的 HTML Body 会交由 SMTP 协议套接字推送至各个目标的收件箱中。
