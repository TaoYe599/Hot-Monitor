## 一、用户画像与核心场景

### 1.1 目标用户

本产品的核心用户分为两大群体：

**个人用户**
- AI 领域研究者、技术爱好者
- 需要追踪特定 AI 模型（如 DeepSeek、Llama）动态的个人开发者
- 希望每天早上能收到一份 AI 热点简报的知识工作者

**团队/企业用户**
- 技术决策者（CTO）、AI 算法研究员
- 市场与竞对分析师
- PR / 品牌监控团队
- 需要多角色协作、多收件人路由的企业情报团队

### 1.2 核心使用场景

| 场景 | 用户 | 触发方式 | 通知方式 |
|------|------|----------|----------|
| 精准关键词监控 | 个人开发者 | 定时轮询（可设置间隔） | 实时邮件推送 |
| 主题热点聚类 | 研究员/分析师 | 定时轮询 | 实时 + 定时简报 |
| 舆情预警 | PR 团队 | 高热阈值触发 | 实时邮件（强穿透） |
| 每日情报汇总 | 所有用户 | 每日 09:00 定时 | 邮件简报 |
| 热点演进追踪 | 技术决策者 | 分数上涨 / 官方背书 | 追加演进邮件 |

## 二、功能点重复性审查

### 2.1 结论：存在 3 处需要关注的重复

#### 重复一：监控模式的"双重入口"导致困惑

当前系统存在两种监控模式（`keyword` / `topic`），但它们的通知链路存在重叠：

```
关键词命中事件
    ├── 触发路径 A：scan-runner → notifyEvent → monitor.notifyChannels（直接发邮件）
    │                 ▲ 走的是 MonitorRecord.notifyChannels，不经过订阅规则
    │
    └── 触发路径 B：scan-runner → 创建事件 → notifyHotspot → dispatchSubscription（订阅分流）
                    ▲ 走的是 SubscriptionRuleRecord 的订阅规则

主题热点聚类
    └── 只走路径 B（订阅规则）
```

**问题**：关键词模式同时走了两条路径，导致：
- Monitor 的 `notifyChannels` 配置和 Subscription Rule 的收件人可以完全不同
- 用户在 Monitor 设置了邮件通知，却又被 Subscription Rule 重复发送到另一批邮箱
- 或用户在 Monitor 设置了通知，但没配置 Subscription Rule，导致该收的人没收到

#### 重复二：Monitor 通知通道 vs. Subscription Rule 收件人的关系不清晰

代码中有两套通知配置：

1. `MonitorRecord.notifyChannels` — 监控任务级别的通知通道（始终走"直接邮件"）
2. `SubscriptionRuleRecord.recipients` — 订阅规则级别的收件人（走"订阅分流引擎"）

两者都指向邮件通知，但彼此独立，没有任何互斥或关联机制。在前端界面中，它们也分散在不同 Tab 中（Monitor Tab 和 Settings Tab），用户很难理解"我应该用哪一套"。

#### 重复三：静默期释放在 Scheduler 和 NotificationService 中各写了一遍

静默队列的释放逻辑（08:00 批量发信）在两个地方都有实现：
- `scheduler.ts` 的 `subscriptionTick()` 方法（08:00 时间点触发）
- `notification-service.ts` 的 `dispatchSubscription()` 方法（22:00-08:00 静默期入队）

这导致静默期逻辑分散，维护成本高。

## 三、功能点安排的合理性评估

### 3.1 整体架构评价

系统分层清晰，数据流合理：

```
配置监控任务 → 调度器触发 → 多源采集 → AI 验证 → 事件/热点 → 通知分发
     ↑                                                              ↓
  Web UI ← SSE 实时推送 ← ← ← ← ← ← ← ← ← ← ← ← ← ← 邮件/推送
```

但有以下安排不合理的地方：

### 3.2 不合理之处

#### 问题 1：Monitor（监控任务）与 Subscription Rule（订阅规则）的职责边界模糊

Monitor 的定位应该是"数据采集配置"，而非"通知配置"。当前的 Monitor 包含了 `notifyChannels`，让它同时承担了"采什么"和"通知谁"两个职责。

**建议**：Monitor 只负责采集配置（关键词/主题、数据源、轮询间隔），通知分发完全由 Subscription Rule 负责。这样职责清晰，用户理解成本低。

#### 问题 2：订阅规则的"定时抓取时机"机制缺失

你的核心诉求是：「早上 9 点发送 → 系统在 8:50 就开始抓取热点数据 → 9 点准时发送汇总」

**当前实现**：
- Scheduler 每 30 秒轮询一次
- 定时简报在到达 `deliveryTime`（如 09:00）时，查询**过去 24 小时**内的所有热点进行汇总
- 也就是说，09:00 发送的邮件包含的是"过去 24 小时的全部热点"，而非"过去 10 分钟内抓取的新热点"

**核心差距**：当前系统在 09:00 只是"查询和汇总"，并不在 08:50 提前触发一轮"专门为这次简报准备的新数据抓取"。所有数据都是监控轮询时陆续采集的，09:00 简报的内容取决于过去 24 小时内轮询积累了多少热点。

如果用户期望的是"每天早上 9 点，准时发起一轮针对特定关键词的新数据抓取，并立刻汇总发送"，当前系统**没有这个能力**。它只能在固定轮询间隔的基础上做汇总。

**建议**：在 Subscription Rule 中增加一个选项："定时简报前 N 分钟预抓取"。开启后，在发送时刻之前，自动触发一次针对该规则所关联关键词的定向数据抓取，保证简报内容是最新的。

#### 问题 3：事件（Event）与热点（Hotspot）的生命周期管理缺失

- Event 是关键词命中的单条记录，可以独立存在
- Hotspot 是 Event 的聚类分组

但没有"清理机制"：旧的 Event 和 Hotspot 永远留在数据库里，只会通过分页展示。数据库会随时间无限膨胀，但没有 TTL 过期策略或自动归档机制。

## 四、热点计分模型评估

### 4.1 当前模型

热点综合分（`score`）的计算由 AI 模型的 prompt 指令驱动，AI 输出 `score` 后直接入库。Prompt 中的评分标准为：

```
score = 信任分(trustScore) × 0.4 + 互动分(engagementScore) × 0.3 + 新鲜分(freshnessScore) × 0.3
```

### 4.2 评分因子分析

| 因子 | 权重 | 数据来源 | 合理性 |
|------|------|----------|--------|
| 信任分（trustScore） | 40% | 来源域名映射（GitHub=0.92, 知乎=0.8, 百度=0.6） | ⚠️ 粗糙，依赖硬编码域名映射，缺乏动态校准 |
| 互动分（engagementScore） | 30% | Twitter 点赞/转发/浏览，HN 点数，Reddit 评分等 | ✅ 跨平台聚合逻辑合理 |
| 新鲜分（freshnessScore） | 30% | 发布时间衰减曲线（3h=1.0, 24h=0.64, 72h=0.48） | ✅ 衰减曲线合理，但缺少周末/节假日特殊处理 |

### 4.3 计分模型的问题

**问题 A：信任分依赖硬编码域名列表（不合理的黑盒）**

`scoreDomainTrust()` 函数使用 `if-else` 硬编码域名映射，新增一个来源需要改代码。这在数据源扩展性上很差。

**问题 B：互动分的归一化标准不统一**

- Twitter 互动分：`(likes × 0.0025) + (retweets × 0.01) + (views × 0.00002)`，点赞权重最低
- HN 互动分：`(points × 0.001) + (comments × 0.002)`
- Reddit 互动分：`(score × 0.0008) + (comments × 0.002)`

Twitter 上一个 1000 赞的帖子得分约 2.5，而 HN 上一个 200 票的帖子得分约 0.2。不同平台的互动分完全不在同一个量纲上，导致跨平台聚类时 hotspot 的 `engagementScore` 失真。

**问题 C：Heuristic Fallback 评分公式与 AI 评分公式不一致**

当 AI 服务不可用时，使用启发式降级公式：

```
score = trustScore × 0.45 + engagementScore × 0.25 + freshness × 0.3
```

与 AI 模式下的公式 `(trust × 0.4 + engagement × 0.3 + freshness × 0.3)` 权重不同，导致 AI 模式和 Heuristic 模式下，同一条内容可能得到不同的排序结果。这对用户体验是困惑的。

### 4.4 建议优化

1. **统一跨平台互动分归一化**：引入平台难度系数，将各平台互动数据换算为统一的"相对热度"（例如：超过该平台 80% 内容的互动量 = 1.0）
2. **动态信任分**：不要用硬编码域名，而应该基于历史数据（用户反馈）持续校准每个域名的信任权重
3. **统一评分公式**：确保 AI 模式和 Heuristic 降级模式使用完全相同的权重比例

## 五、邮件格式评估与你的核心诉求分析

### 5.1 当前邮件格式现状

系统已实现两类 HTML 邮件模板（均在 `notification-service.ts` 中）：

#### 实时预警邮件（`renderInstantAlertEmail`）
- ✅ 包含：综合热度、新鲜度、互动度三环指标
- ✅ 包含：AI 摘要、信源链路与信任分、可点击正/负反馈按钮
- ✅ 视觉：Apple 毛玻璃风格，卡片式布局
- ⚠️ 问题：布局较扁平，卡片层级感不强

#### 周期简报邮件（`renderPeriodicDigestEmail`）
- ✅ 包含：TOP 3 高热度排行榜、主题卡片流、洞察看板
- ✅ 包含：关联事件摘要、信源标签
- ⚠️ 问题：TOP 3 排行榜只渲染分数，没有"变化趋势"（PRD 中提到但未实现）

### 5.2 关于你的核心诉求："早上 8:50 预抓取，9:00 发送"

经过代码分析，当前系统的定时简报机制是：

```
09:00 触发
    → 查询过去 24h 的所有热点
    → 按规则过滤匹配
    → 渲染并发送邮件
```

**不是**你说的"8:50 预抓取"。所有数据都是监控轮询（默认 15 分钟一次）陆续积累的，09:00 简报只是查询和汇总。

**如果你需要"每天早上 9:00，精准抓取最新数据并发送"：**

**方案 A（推荐）**：在订阅规则中增加"定时预抓取"选项

- 用户配置"每天 09:00 发送简报"
- 系统在 08:50 自动触发一轮针对该规则关键词的定向数据抓取
- 09:00 将这批新数据连同历史热点一起渲染发送

这需要在 Scheduler 中增加一个"简报预抓取队列"，或者在 Subscription Rule 增加一个 `prefetchBeforeMinutes` 字段（如 `10`，表示发送前 10 分钟预抓取）。

**方案 B（简单替代）**：缩短监控轮询间隔

- 将 `intervalMinutes` 从默认 15 分钟改为 5 分钟
- 这样 09:00 发送时，数据库中至少已有过去 5 分钟内采集的数据
- 不需要改架构，但抓取时机仍然不精确

### 5.3 邮件格式的改进建议

基于 PRD 中的规范和当前实现，建议以下改进：

1. **实时预警邮件**：增加"信源类型标签"（如 [官方机构]、[技术社区]、[海外高分社区]），方便收件人一眼辨别信息权威性
2. **周期简报邮件**：增加"变化趋势箭头"（↑ 新增 / → 维持 / ↓ 下降），让收件人快速感知热点是在升温还是降温
3. **零警报安全信**：当前已实现（无匹配热点时发送"今日平稳"邮件），保持即可
4. **邮件标题优化**：当前标题如 `[⚡ 实时预警] Hot Monitor 重大发现: DeepSeek-v4...`，建议增加收件人姓名占位符，增加个性化感

## 六、总结与优先级建议

### 6.1 功能优先级

| 优先级 | 问题 | 影响 | 建议 |
|--------|------|------|------|
| **P0** | Monitor 与 Subscription Rule 职责重叠 | 用户困惑，可能导致漏发或重复发 | 明确职责边界，建议 Monitor 只管采集 |
| **P0** | 定时简报"预抓取"机制缺失 | 无法实现"早上 9 点准时抓取发送"的场景 | 增加订阅规则预抓取选项 |
| **P1** | 跨平台互动分归一化缺失 | 热点排序失真，跨平台热点不公平比较 | 引入平台难度系数归一化 |
| **P1** | 静默期逻辑分散在两处 | 维护成本高，08:00 释放逻辑重复 | 统一到 NotificationService |
| **P2** | 信任分硬编码域名映射 | 可扩展性差 | 改为基于历史反馈的动态信任分 |
| **P2** | 缺少 Event/Hotspot 生命周期管理 | 数据库无限膨胀 | 增加 TTL 过期或归档机制 |
| **P3** | Heuristic 与 AI 评分公式不一致 | 降级时热点排序跳跃 | 统一公式权重 |

### 6.2 当前系统成熟度评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 核心采集能力 | ⭐⭐⭐⭐⭐ | 多源采集覆盖全面，RSS/Feeds/GitHub/HN/知乎/Reddit 等 |
| AI 判重与聚类 | ⭐⭐⭐⭐ | OpenRouter 结构化输出 + 降级机制健全 |
| 通知分流引擎 | ⭐⭐⭐ | 订阅规则设计完善，但 Monitor/Rule 职责重叠 |
| 邮件格式 | ⭐⭐⭐ | 基础模板完善，但缺少趋势线和个性化 |
| 定时简报机制 | ⭐⭐ | 缺少"预抓取"机制，无法精准控制发送时机 |
| 前端交互体验 | ⭐⭐⭐⭐⭐ | UI 精致，Apple 毛玻璃风格，用户体验优秀 |
| 计分模型 | ⭐⭐⭐ | 功能可用但跨平台归一化缺失 |
